## Key finding

The **only** non-200 Slack received is the **`416 Range Not Satisfiable`**
from `Slackbot-LinkExpanding`. Two of its requests landed in the same second
from different AWS IPs; one got `206 Partial Content` (success), one got
`416`. nginx serves static files with `Accept-Ranges: bytes`, so Slack is
sending a `Range:` header; a 416 means the requested range start was at/Beyond
the file length at that instant.

Plausible explanations to test (none confirmed):

1. **Range-request handling** — Slack's validator/unfurler sends a Range that
   nginx rejects as 416 for this static file. A cheap server-side mitigation
   is to disable range support for the feed (`max_ranges 0;` in the
   `location`), forcing a plain `200` every time.
2. **Content-Type pickiness** — feed is served as `text/xml`; the
   standards-correct type is `application/rss+xml`. Some readers are strict.
   Cheap to add via an `application/rss+xml xml;` types override or a
   `location = /ai-digest/feed.xml { default_type application/rss+xml; }`.
3. **Cached add-time failure** — if the very first `/feed subscribe` ran
   before nginx/cert was fully live (vhost symlink + cert both date to
   Jun 9), Slack may have recorded a failure. Fix: `/feed remove …` then
   re-subscribe now that it's confirmed live.
4. **Work-workspace specific** — different egress/proxy or a workspace policy
   on the work Slack. This is why the user is retrying from the work computer
   first.

## What is NOT done

1. **Root-cause the 416 / Slack failure.** Not reproduced against Slack
   directly (we can't drive their `/feed` app). The 416 is a correlation, not
   a proven cause.
2. **No mitigation applied.** Candidate fixes 1–2 above (max_ranges 0;
   application/rss+xml) are NOT made — they need an nginx edit + `sudo nginx
   -t && sudo systemctl reload nginx` (privileged), and the user wanted to
   table it and test from work first. Do not apply without the user's go.
3. **HTTP→HTTPS redirect** — minor, unrelated to Slack (Slack uses https):
   the vhost's plaintext server listens on **port 8080, not 80**, and there
   is no port-80 redirect block. `http://feeds.elucia.com/…` (port 80) hits a
   different default server and 404s. Not the cause; note only.
